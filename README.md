# OpenClaw Docker Infrastructure

This repository defines the infrastructure, deployment configurations, and secure sandbox environments for hosting **OpenClaw**. It uses a multi-layered containerized architecture to provide high security out-of-the-box, ensuring robust isolation between the control-plane gateway, task workers, and task sandboxes.

## Requirements

To run this infrastructure successfully, your host system must meet the following requirements:

- **OS**: Linux or WSL2. (Mac can build images if they want, it's not tested)
- **Hardware**:
  - Minimum 4GB RAM (8GB+ recommended for browser-based tasks).
  - Prebuilt docker images are for `amd64` (x86_64) and `arm64` (AArch64).
- **Docker**: Docker and Docker Compose.
- **Sysbox (Optional)**: If you use [Sysbox](https://github.com/nestybox/sysbox), the node runs rootless. On WSL2 or if Sysbox is missing, the `setup.sh` script automatically falls back to standard `runc` + `privileged` mode. Since all agents run in their own containers _inside_ the node, this fallback remains secure.
- **Cloudflare Tunnel (Optional)**: Recommended for users who want to safely expose their dashboards to the internet via their own domain name without opening firewall ports.

## Getting Started

1. **Initialize the Environment**:
   Run the setup script from the repository root to populate required variables into `./compose/.env`.

   ```bash
   ./setup.sh
   ```

2. **Configure your Environment**:
   Open `compose/.env` and add any necessary AI model API keys (e.g. `OPENROUTER_API_KEY`, etc.). This file was created during the previous step by the `setup.sh` script.

3. **Launch the Core Infrastructure**:
   Venture into the `compose/` directory to spin up the gateway, secure node, and networking.

   ```bash
   cd compose
   docker compose up -d
   ```

## Architecture & Setup Overview

The standard deployment configuration runs on the host using Docker Compose and orchestrates three primary services:

1. **OpenClaw Gateway**: Acts as the central control plane, receiving tasks and orchestrating agents.
2. **OpenClaw Secure Node (`openclaw-node`)**: A privileged systemd-based container acting as an independent, highly isolated virtual machine. It runs its own internal Docker-in-Docker daemon, networking stack, and security proxies. Inside this environment, it mounts a nested docker-compose setup to launch the **OpenClaw Node Host** (a worker), which registers back to the Host Gateway.
3. **Cloudflare Tunnel (Optional)**: Securely exposes the node/gateway services over the web without having to expose host firewall ports or manage complex SSL configurations. This is only needed if you want remote access through your own domain.

Through this structure, tasks sandboxed by the OpenClaw node are executed within a Docker daemon residing securely inside an isolated, containerized Ubuntu environment that is monitored by custom security daemons like Predicate Authority and Mitmproxy.

## Published Images

This project builds and publishes four primary Docker images to the GitHub Container Registry:

- **`openclaw-gateway`**: The central brain of the system. It handles user authentication, task orchestration, and manages the state of all connected nodes.
- **`openclaw-node`**: A specialized, secure Ubuntu-based environment designed to host OpenClaw workers. It features an internal systemd init, a dedicated Docker-in-Docker (DinD) daemon, and pre-integrated security sidecars (Predicate Authority and Mitmproxy) for traffic filtering and policy enforcement.
- **`openclaw-sandbox`**: A lightweight, ephemeral execution environment where AI agents actually run their code. It is designed for maximum speed and security, providing only the necessary runtime tools without persistent storage.
- **`openclaw-sandbox-browser`**: An extension of the standard sandbox that includes a pre-installed headless browser and necessary drivers for web-based automation tasks.

## Configuration Files

The security and behavior of the **OpenClaw Node** are managed through several key configuration files located in the `compose/` directory:

- **`.env`**: (Generated by `setup.sh`) Contains sensitive environment variables, tokens, and runtime toggles for the entire stack. You will want to add your AI model API keys (e.g. `OPENROUTER_API_KEY`, `OPENAI_API_KEY`, etc.) directly to this file. The `openclaw-gateway` is configured to load these automatically.
- **`policy.yaml`**: The security policy for the Predicate Authority sidecar. It defines which actions (filesystem, network, shell) agents are allowed to perform. It uses a "secure-closed" strategy where anything not explicitly allowed is blocked.
- **`mitm_filter.py`**: A Python script for Mitmproxy that handles real-time traffic filtering and automated token injection for external APIs (like OpenRouter, Brave Search, etc.).
- **`predicate-authorityd.toml` (Optional)**: The core configuration for the Predicate daemon, managing its internal settings and how it interfaces with the node.
- **`iptables-rules.sh` (Optional)**: A shell script used by the `iptables-loader` service to set up firewall rules and transparently proxy traffic through the security sidecars. You probably don't need to change this.
- **`node-docker-compose.yml` (Optional)**: The Docker Compose configuration that runs _inside_ the node container to manage the OpenClaw worker and its registration with the host gateway.

## Repository Layout

- `compose/`
  The main deployment directory. Contains the primary `docker-compose.yml` for the host, along with `.env` configurations. It also houses `node-docker-compose.yml`, which is safely mounted into the `openclaw-node` service and acts as the setup for the nested, internal worker daemon.
- `node/`
  Contains the Dockerfile, systemd definitions, and `install.sh` needed to build the `openclaw-node` image. This image bundles Docker, Sysbox, Predicate Authority, and Mitmproxy to create the resilient Docker-in-Docker system layer.
- `sandbox/`
  Contains definitions and install scripts for the lightweight execution sandbox images that OpenClaw will use to actually perform tasks securely.
- `scripts/`
  Houses utility scripts relevant to building or lifecycle hooks for the node.
- `setup.sh`
  An interactive setup script that helps bootstrap your `compose/.env` file. It automatically discovers and injects the host's Docker socket GID and generates secure tokens (like `OPENCLAW_GATEWAY_TOKEN`) required for components to communicate.
